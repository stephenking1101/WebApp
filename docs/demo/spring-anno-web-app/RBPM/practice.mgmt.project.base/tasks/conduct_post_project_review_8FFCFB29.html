Task: Conduct Post Project Review

Purpose

Assess the extent to which a project was successfully executed - that is, how efficient and effective the project and project management processes were, and how well they were implemented.

Procedure

To perform a post project review:

  1. Analyse the project documentation
  2. Conduct interviews with project team members and others as required (see Interview Guidelines below)
  3. Document your results in a Post Project Review report. Your report should highlight areas that indicate the need for action such as:
    • Process modification
    • Retraining
  4. Distribute your report to all the key stakeholders 

Project Documentation Review Guidelines

  • Check that all documents are in place. If any documents are missing, check that an acceptable justification was provided.
  • Check that documents conform to standards. If any deviations are noted, check that appropriate authorisation was obtained.
  • Check that all activity completion criteria have been met and documented.
  • Check that issue and change management logs have been completed.

Interview Guidelines

There are two main objectives of the interviews:

  • To solicit the opinions of the interviewees about the effectiveness of the development processes
    • How closely did they follow the processes?
    • Did the defined processes help or hinder the development?
    • Do they have any suggestions for modifications to the processes?
  • To gather additional information about any anomalies you have identified from your documentation reviews or from previous interviews

Relationships

Outputs
  • Post Project Review

    Steps

    Set-up

    The Post Project Review may involve:

    • examination of project documents
    • interviews with project team members
    • interviews with organisations that supported the project.

    To set up a Post Project Review:

    1. Identify individuals whom you wish to interview
    2. Establish timescales for the review (see Guidelines below)
    3. Define the distribution list for the final report. This will normally consist of:
      • Project Sponsor
      • Management
      • Budget Owner.

    Guidelines

    The timing of the review needs careful judgement. You will need some time to allow users of the system to give you considered opinions about the product, rather than first impressions. But you also need to hold the review while the development is still fresh in the memories of the project team, and before the project team is widely dispersed on other assignments.

    Initiate

    The normal sequence of a Post Project Review is to analyse the project documentation, then conduct interviews to explore particular areas in more detail. To initiate the post project review:

    1. Arrange access to the project documentation.
    2. Schedule interviews with appropriate project team members (see Guidelines below). Allow sufficient lead time to enable you to perform a preliminary analysis of the project documentation.

    Guidelines

    You choose interviewees appropriate to the project. A typical list would include:

    • Project Sponsor
    • Project Manager
    • HOST IT Operational Staff
    • Test Manager
    • User Representative.
    Perform Review

    To perform a post project review:

    1. Analyse the project documentation (see Project Documentation Review Guidelines below).
    2. Conduct interviews with project team members and others as required (see Interview Guidelines below).
    3. Document your results in a Post Project Review report. Your report should highlight areas that indicate the need for action such as:
      • Process modification
      • Retraining
    4. Distribute your report to the list identified in the procedure Set-up Post Project Review

    Project Documentation Review Guidelines

    • Check that all project documents are in place. If any documents are missing, check that an acceptable justification was provided.
    • Check that documents conform to standards. If any deviations are noted, check that appropriate authorisation was obtained.
    • Check that all activity completion criteria have been met and documented.
    • Check that issue and change management logs have been completed.
    • Check at what activities risk reviews have been performed, and check that risks have been managed from activity to activity.

    Interview Guidelines

    There are two main objectives of the interviews:

    • To solicit the opinions of the interviewees about the effectiveness of the development processes
      • How closely did they follow the processes?
      • Did the defined processes help or hinder the development?
      • Do they have any suggestions for modifications to the processes?
    • To gather additional information about any anomalies you have identified from your documentation reviews or from previous interviews

    Post Project Review document guidelines

    1. Ensure that all applicable sections of the Introduction and Artifacts sections are complete.
    2. In the Process Experience sections record any findings from the review according to the phase in which it occurred. Findings are recommended to be organised into the following categories, Description, Findings and Recommendations.
    3. Document the appropriate sections of Solutions effectiveness.
    4. Record your significant recommendations and best practices from the Process Experience sections in the Summary of recommendations / Lessons Learned matrix.
    5. Record any action items identified during the Post Project Review in the Actions identified from the review matrix. Each action should have an owner and a priority.
    Process report

    To process the report, act on recommendations in the Post Project Review. This could include:

    • Improvements to the project outputs such as user or operation documentation.
    • Additional staff training. For example, if planning was inaccurate, would the Project Manager benefit from additional training?
    • Changes to the development processes
