Task: Define / Validate Scope

Purpose

Defining / validating the scope of a project develops a common understanding of what is included in and excluded from the project. The scope builds upon the project concept developed in the Initiation Stage.

Steps

Identify and clarify the project purpose / business need
  • Check to see if the purpose / business need was addressed in the Initiation Stage.

You may need to work with the project sponsor or other project stakeholders to get this information if not available from the Initiation Stage.

  • Write an overview of the purpose of the project.
  • Specify the business value or benefits to the organisation.
Define project goals

Define measurable benefits / goals in terms of how and when they can be measured (see Guidelines below).

Identify the business benefits / goals that will accrue from the successful completion of the project and define these by how they will be measured, and when they can be measured. For example:

The project will return a net revenue increase of $200,000 within 12 months of implementation.

You may find it useful to apply SMART criteria to your objectives. SMART objectives are:

Specific

Clear about what is changed, where, when, and how.

Measurable

Quantified targets and benefits that can be measured against a current baseline.

Achievable

Achievable within known constraints of resource and environment.

Relevant

Reflecting what stakeholders need and want.

Time bound

Constrained by time or time-related dependencies.



Identify In-scope and Out-scope

Describe what the project will accomplish – in scope – and what the project will not accomplish – out of scope.

In-scope is the all work and only the work that is required to be perform to achieve project objectives.

Out-scope is work is required for the result of the project to be successful, but excluded from the current scope of the project.

Define project constraints, assumptions and dependencies

Constraints

Describe the real-world limits for the project. Constraints are usually mandated, unavoidable, or may prevent delivering the project in the most appropriate way. Types of constraints are budget, schedule, staff, equipment, facilities and infrastructure, etc.

Assumptions

Identify items taken as being true for the purposes of planning a project but could change later. Indicate the potential impact if the assumptions prove to be false.

Dependencies

Describe the critical dependencies which could potentially induce an impact, such as causing the project to miss the end date to the project.

Identify key project deliverables
Describe high-level key deliverables. A deliverable is a measurable, tangible, verifiable outcome, result, or item that must be produced to complete a project or part of a project.

Key Considerations

While defining the project goals, constraints and assumptions, refer to Considerations for assessing impact to identify any elements which could impact (positively and negatively) the sustainability goals.
